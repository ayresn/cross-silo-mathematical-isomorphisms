# ADVERSARIAL REVIEW PROTOCOL: SID ENTRY PRE-VALIDATION (Stage 2)

This file contains the standardized prompt used to run cross-lab adversarial consistency review on candidate SID entries before they advance to Stage 3 human bibliometric validation. To preserve the value of the adversarial check, the reviewing model should be from a **different AI provider than the model that generated the entry** — same-provider models are more likely to share blind spots from similar training distributions.

> **Pipeline context:** This protocol governs Stage 2 (adversarial pre-validation) only. Its scope is deliberately limited to internal consistency and face-validity — checks that require only the entry text itself, answerable without external tools or literature access. Stage 2 answers: **"Does this entry say what it claims to say?"** Stage 3 (human bibliometric validation) answers the separate question: **"Is this claim novel and does it survive expert scrutiny?"** Do not conflate the two.

> **What this stage is not:** Stage 2 is not a gatekeeper that prevents bad entries from being committed to the repository. Entries that fail Stage 2 are still committed — as `adversarial-rejected` — so that the pipeline's false-positive rate remains computable and near-miss entries retain their negative scientific value. Stage 2 is a **triage filter**: it prevents Stage 3 human effort from being spent on entries with internal consistency failures that require no literature access to catch.

> **Note on automated pre-checks:** A lightweight lint script (checking that `entry_id` matches the filename number, that `triple_correspondence_vectors` contains exactly 3 items, that `maturity_stage` is set to `"candidate"`, and that `generation_timestamp` is a plausible date) should be run before invoking this protocol. Those structural checks are cheaper than a model call and should not consume adversarial review capacity. This protocol covers semantic and mathematical consistency only.

Paste the full candidate entry below this line and submit to the reviewing model:

````text
You are acting as an adversarial Stage-2 reviewer for the Cross-Silo Mathematical Isomorphisms repository. The candidate entry below was generated by a different AI model. Your task is to evaluate its internal consistency and face-validity before it advances to Stage 3 human bibliometric review.

YOUR SCOPE IS LIMITED TO:
Internal consistency and face-validity — does this entry say what it claims to say, determined by reading the entry text alone. No external tools, no literature search, no bibliometric analysis. That is Stage 3's job.

YOUR SCOPE EXPLICITLY EXCLUDES:
- Searching for or citing external literature
- Judging whether the isomorphism is novel against the published record (Stage 3)
- Rewriting, improving, or correcting the entry — flag it, do not fix it

ADVERSARIAL STANCE:
You are looking for specific, quotable failures — not stylistic preferences, not subjective judgment calls within a reasonable range. A FAIL verdict requires you to quote the specific failing text from the entry and state precisely what is wrong with it. Do not issue FAIL verdicts for things that are merely imprecise, compressed, or less rigorous than you would prefer — reserve them for genuine errors: wrong equations, category mismatches, contradictions between the YAML and body, textbook analogies the rejection criteria were designed to catch.

---

CHECK 1 — YAML METADATA INTEGRITY
Verify:
- `providence.model_family` and `providence.model_version` are plausible AI model identifiers (not framework names, dataset names, or the string "SID")
- `generation_timestamp` is a plausible recent date (not years in the past or future)
- `triple_correspondence_vectors` lists exactly 3 distinct items
- `maturity_stage` is set to `"candidate"`
- `relationship_type` is `"candidate_structural_isomorphism"`

CHECK 2 — EQUATION VALIDITY
For each equation displayed in Section 3:
- Does it actually model what the accompanying Silo A / Silo B description claims it models?
- Are the variables, operators, and equation type (parabolic/elliptic/hyperbolic/stochastic) consistent with the stated physical domain?
- Is the equation genuinely from the stated domain, or is it a standard equation from a third field being misattributed or relabeled?
- Do the two equations together actually support the structural isomorphism claimed in Section 1, or does one or both fail to demonstrate the correspondence under examination?

CHECK 3 — VOCABULARY MATRIX COHERENCE
For each paired mapping in Section 2 (Silo A Token ↔ Silo B Token):
- Are the two terms objects of compatible mathematical type? (Both operators, both scalar state variables, both dimensionless parameters, etc.) A spatial domain mapped to a point in time, a physical continuum quantity mapped to an administrative policy threshold, or a rate mapped to a position are category errors, not judgment calls.
- Does the Operator Role explanation actually explain a mathematical equivalence, or does it only assert similarity using hedged language ("analogous to," "plays a similar role as") without specifying the shared mathematical structure?

CHECK 4 — TRIPLE-CORRESPONDENCE BODY VERIFICATION
The YAML `triple_correspondence_vectors` lists three claimed correspondences. For each one, verify that Section 3's body text discusses it with mathematical specificity — not merely uses related vocabulary, but actually demonstrates the correspondence with an equation, operator, or derivation. A YAML field checked off with no supporting body text is a FAIL. Partial coverage (body text gestures at the concept without demonstrating the correspondence) warrants FLAG.

CHECK 5 — REJECTION CRITERIA FACE-CHECK
From your training knowledge alone (no literature search):
- Is this domain pairing a canonical interdisciplinary analogy already recognizable from graduate-level textbooks or widely-cited review articles? Examples the protocol explicitly rejects: Schrödinger ↔ paraxial wave optics, heat ↔ solutal diffusion, Ising model ↔ lattice gas. If the entry's pairing is comparably well-known, issue a FAIL with the textbook or review source you recognize it from.
- Is the methodological transfer in Section 4 genuinely asymmetric? If the transfer could plausibly go in either direction with comparable benefit, the asymmetry criterion is not satisfied — issue a FLAG.
- Is Section 4's falsifiable prediction actually falsifiable? Does it name a specific, measurable outcome that would differ from current practice? A prediction that amounts to "this might work better" is not falsifiable — issue a FLAG.

CHECK 6 — SCORE-CONTENT PLAUSIBILITY
Review the `prior_discovery_metrics` scores in the YAML against what the entry's body actually demonstrates. Flag any score that is obviously inconsistent with the content — particularly:
- A high `structural_isomorphism_score` (≥7.0) paired with an equation in Section 3 that does not actually demonstrate the claimed isomorphism
- A high `operator_equivalence_confidence` (`high` or `very_high`) paired with a vocabulary matrix that contains category errors
- A `representation_mismatch_score` that appears significantly inflated given how closely related the two domains' foundational objects actually are

Do not penalize scores for being in a plausible range even if you would score differently — only flag obvious contradictions between the score and the content.

---

OUTPUT FORMAT:

Produce output in exactly three labeled blocks, in this order, separated by the exact divider lines shown. Do not add any text outside these blocks.

--- BEGIN YAML AMENDMENTS ---

Output only the YAML fields that need to change. Always include `maturity_stage` (at top level, under `sid_metadata`) and a new `adversarial_review` block nested inside `validation_status`. Reproduce unchanged fields only if needed for YAML structural context.

validation_status:
  adversarial_review:

    review_timestamp: "[YYYY-MM-DD]"
    verdict: "[PASS / FLAG / REJECT]"
    verdict_rationale: "[One sentence stating the primary basis for the verdict]"
    failed_checks: ["[Check N: brief description]"]  # empty list [] if none failed
    flagged_checks: ["[Check N: brief description]"]  # empty list [] if none flagged
    stage_3_watch_items: ["[item]"]  # specific things for the human reviewer to probe even if passing; empty list [] if none

--- BEGIN ENTRY SECTION 6 (ADVERSARIAL REVIEW) ---

Output a Section 6 block formatted exactly as shown, to be appended to the entry's .md file:

- **CHECK 6 (Score-Content Plausibility):** [PASS / FLAG / FAIL] — [If FAIL or FLAG, name the specific score and the content it contradicts.]

[Bullet list of specific things the human reviewer should probe at Stage 3 even if this entry passed adversarial review. If none, write "None identified."]

--- BEGIN README DIRECTORY SNIPPET UPDATE ---

Output the updated four-line README directory entry with the status tag updated to reflect the adversarial verdict. Copy the System Synthesis, Domains, and Isomorphism lines unchanged from the original entry; only the status tag changes:

*   **[filename.md](path/filename.md)** — `[Stage 2 / cleared | Stage 2 / flagged | Stage 2 / rejected]`
    *   *System Synthesis:* [unchanged from original entry]
    *   *Domains:* [unchanged from original entry]
    *   *Isomorphism:* [unchanged from original entry]

---

VERDICT LOGIC:
- PASS → adversarial-cleared: All 6 checks pass with no flags. Entry proceeds to Stage 3 queue.
- FLAG → adversarial-flagged: One or more FLAG-level issues (implausible score, partial body support, weak falsifiability, asymmetry concern) but no fatal flaws. Entry proceeds to Stage 3 with watch items noted for the human reviewer.
- REJECT → adversarial-rejected: Any FAIL-level issue: wrong or misattributed equation, category-error vocabulary mapping, all three YAML vectors unaddressed in body, recognizable textbook analogy, or YAML metadata containing non-model identifiers. Entry does NOT proceed to Stage 3. It is committed to the repository as adversarial-rejected for false-positive-rate tracking, but is clearly marked as not requiring Stage 3 human effort.
````

## MAINTAINER WORKFLOW

After receiving the three output blocks from the reviewing model:

1. **Apply YAML amendments:** Merge the `--- BEGIN YAML AMENDMENTS ---` block into the entry's frontmatter. The `maturity_stage` field moves from `candidate` to `adversarial-cleared`, `adversarial-flagged`, or `adversarial-rejected`. The new `adversarial_review` sub-block is added inside `validation_status`.

2. **Append Section 6:** Copy the `--- BEGIN ENTRY SECTION 6 ---` block to the end of the entry's `.md` file. This is the first content appended to the entry file itself (Sections 1–5 are generated at Stage 1; the original Section 6 output — the README snippet — was never part of the entry file).

3. **Update the README directory:** Replace the entry's existing `Stage 1 / pending` status tag in the CORE DATASET DIRECTORY with the updated tag from the `--- BEGIN README DIRECTORY SNIPPET UPDATE ---` block.

4. **Commit:** Commit the updated entry file and README in a single commit. For `adversarial-rejected` entries, add a commit message noting the rejection so it is visible in the commit log.

5. **Stage 3 queue:** Only entries at `maturity_stage: adversarial-cleared` or `adversarial-flagged` are added to the Stage 3 human bibliometric review queue. Entries at `adversarial-rejected` are retained in the repository for false-positive-rate tracking but require no further action unless the rejection is appealed and the entry is regenerated.
